# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

kurly is a Jsonnet library of Kubernetes workload recipes. You start from a base kind (`http`, `worker`, `cron`, `daemon`) and add capabilities as composable `+` features (`kurly.store`, `kurly.config`, `kurly.expose.gateway`, `kurly.security.baseline`, ‚Ä¶) ‚Äî each a `{ config+:: ‚Ä¶ }` mixin, so they late-bind against the merged config in any order. Built on k8s-libsonnet. It publishes as the single-layer OCI image `ghcr.io/metio/kurly` (JOI-shaped: `FROM scratch`, jb-vendor-tree layout under `/github.com/metio/kurly/`), consumable by [jaas](https://github.com/metio/jaas) as a Flux `OCIRepository` source or an image-volume mount (author a workload as `function(params) ‚Ä¶` and JaaS feeds the params as TLAs), and locally via `jb install`.

## Common commands

The toolchain and the dev-task commands are defined once in `flake.nix` and version-pinned by `flake.lock` (Renovate-maintained): CI runs every gate through the flake's devShell, so a gate that is green locally is green in CI by construction. Each multi-step gate is a `writeShellApplication` command (a `scripts/<name>.sh` file wrapped by nix ‚Äî shellchecked at build, hermetic `runtimeInputs`), on `$PATH` inside `nix develop` and callable one-shot. There is no host task-runner (no `just`/Makefile wrapper) ‚Äî the commands live in the shell they run in.

```shell
nix develop --command verify           # every gate, serially ‚Äî the local pre-push check (what CI runs)
nix develop --command check-fmt        # jsonnetfmt --test across all sources
nix develop --command check-tests      # jb install + assertion suite + the requiresService/exclusion negative checks
nix develop --command check-examples   # render examples + workloads, validate with kubeconform
nix develop --command check-security   # kurly's Rego invariants (conftest) + pluto (deprecated APIs) + kubesec
nix develop --command reuse lint       # the single-tool gates call the tool directly:
nix develop --command yamllint .
nix develop --command actionlint       # shellcheck rides in the devShell, so run: blocks are linted too
nix develop --command markdownlint-cli2 '**/*.md' '#vendor'
nix develop --command typos
```

Or drop into the shell ‚Äî `nix develop` prints the command menu on entry, then run `verify` / `check-*` / any tool (`jsonnet`, `jb`, `kubeconform`, ‚Ä¶) bare. Plain `nix-shell` works too (`shell.nix` reuses the flake via flake-compat, pinned through `flake.lock`).

On this host, `nix` is [nix-portable](https://github.com/DavHau/nix-portable) (`~/.local/bin/nix`, store in `~/.nix-portable` ‚Äî user-level, no system install) and needs `NP_RUNTIME=bwrap` (exported in `~/.bashrc`; the auto-selected runtime cannot nest the build sandbox's mount namespace here). A system-level nix install is not possible on this Fedora Atomic host (the transient-root workaround broke boot). From a sandboxed shell, where nix-portable's user namespaces are blocked, resolve the same flake through the nix container image: `podman run --rm -v "$PWD:/work:z" -v metio-nix:/nix -w /work docker.io/nixos/nix:latest` with `NIX_CONFIG='experimental-features = nix-command flakes'`. **direnv is not usable here** ‚Äî nix-portable's store is namespace-local, so an exported devShell env references `/nix` paths the host can't see; `nix develop` is the workflow. A contributor with a real nix install may add a local, gitignored `.envrc` (`use flake`) if they want direnv auto-loading.

jsonnet-lint is deliberately not part of the gate: it cannot resolve the vendored k8s-libsonnet's internal `doc-util` imports (jb `legacyImports: false`) and drowns real findings in vendor noise.

**Kubernetes static analysis (`check-security`)** is weighted toward custom policy. `policy/*.rego` holds kurly's own invariants, run over the RENDERED manifests with **conftest** ‚Äî Rego reads the JSON jsonnet emits, so it enforces structural guarantees the general tools can't express (the managed-by label, tag-pinned images, stable selectors, volume/mount consistency), green by construction since they hold for every workload regardless of composed features. Backing it are two small, fast, no-config tools: **pluto** (removed/deprecated API detection) and **kubesec** (a security risk score with a floor the hardened defaults clear at 7-11). The heavier best-practice scorers (kube-score/kube-linter/trivy) are deliberately NOT used ‚Äî they demand a large ignore-list of the deployment-topology checks a workload-recipe library does not emit, and Rego covers kurly's real contract precisely instead. The security-relaxation examples (`legacy`, `cron`) are exempt from the kubesec score (they exist to demonstrate the escape hatches) but still policy-checked.

**`kind-smoke`** is the real-cluster complement (the `.github/workflows/kind-smoke.yml` gate, self-gated on a diff, not part of `verify` since it needs a cluster). envtest does not fit ‚Äî there is no controller to reconcile; kurly emits manifests, so the thing to prove is that they *run*, not just validate. The workflow spins a throwaway kind cluster (kind + kubectl from the flake, never a marketplace action), and `scripts/kind-smoke.sh` applies the workloads whose images actually exist ‚Äî the nginx examples and the **tik** workload (real published image; the stateful path exercises a WaitForFirstConsumer PVC, a mounted ConfigMap, an optional Secret, a scratch emptyDir, a pinned uid, and Recreate) ‚Äî then waits for their Deployments to become Available. tik's readiness probe is `GET /tickets.edn`, so a green rollout proves the board serves. The placeholder-image examples are rendered/schema-checked by check-examples, not run here.

## Architecture

The library's entry point `main.libsonnet` lives at the **repo root** so the jb import path is `github.com/metio/kurly/main.libsonnet` ‚Äî the same path the OCI image serves via jaas's vendor-tree search, keeping local and in-cluster imports identical. The recipe modules it imports (`base`, `features`, the workload kinds, `expose`, `security`, `migrations`, `k`) live under **`lib/`** and are reached as `import './lib/‚Ä¶'`. The library image therefore ships `main.libsonnet` **and** the whole `lib/` tree ‚Äî an image with only `main.libsonnet` fails the first `import './lib/‚Ä¶'` a consumer triggers (a single COPY of both, staged so the image stays one layer; see the `Containerfile`). Deployable workloads live under **`workloads/<name>/`** as one `<stage>.libsonnet` per stage ‚Äî a `function(params)` returning a COMPOSABLE app (a base with defaults, no exposure), which a consumer imports by canonical path, adapts with `+` features, and renders with `kurly.list`; plus a `README.md` showing the end-to-end deploy (import ‚Üí JaaS `JsonnetSnippet` referencing the kurly `JsonnetLibrary` ‚Üí stageset `StageSet`). Each workload directory is an independently released unit (see CI below), so the library must stay at the root and workloads must stay in their directories.

- `k.libsonnet` ‚Äî the single pin of the k8s-libsonnet API version (a directory like `1.35`); every module imports k8s-libsonnet through this file, so a version bump is one line.
- `base.libsonnet` ‚Äî the shared core: a hidden `config::` object, the manifest fields computed from it (`container`/`podSecurity`/`podVolumes`, the `deployment`/`service`/`cronjob`/`daemonset` bodies, the owned `storeClaim`/`configMap` handles), plus the exclusion-group `assert`. Holds **no** `with*` methods ‚Äî capabilities live in `features.libsonnet`.
- `features.libsonnet` ‚Äî every capability as a `+`-composable function returning a `{ config+:: ‚Ä¶ }` mixin (`kurly.store`, `kurly.config`, `kurly.secretMount`, `kurly.scratch`, `kurly.runAs`, `kurly.recreate`, `kurly.args`, `kurly.env`, `kurly.probes`, `kurly.resources`, `kurly.replicas`, the security hatches, ‚Ä¶). A feature only ever writes `config`, never a manifest, so the computed fields late-bind against the merged config regardless of compose order.
- `http|worker|cron|daemon.libsonnet` ‚Äî one workload kind each, a `function(name, image)` (cron also takes a `schedule`) that composes `base.core` + the kind's mixins/manifests. These are the base "defaults" you add features onto.
- `expose.libsonnet` ‚Äî exposure recipes composed onto a `kurly.http` app with `+`: `ingress` (Ingress API), and for the Gateway API `gateway`/`listenerSet` (attach an HTTPRoute to an existing parent) and `ownGateway`/`ownListenerSet` (generate the parent too). **Workload and exposure are deliberately two separate composable axes ‚Äî do not fold routing back into a workload kind or add a mode toggle.** Every Gateway API recipe emits an HTTPRoute. Each recipe captures its `host` argument lexically. A shared `requiresService` object-level assert fails composition onto Service-less kinds, and **all five join the `exposure` exclusion group** so composing two fails the render (see the exclusion mechanism below). The Gateway API objects are plain manifests (no gateway-api-libsonnet) to keep consumers' render-time dependency closure at k8s-libsonnet alone; ListenerSet is GA (`gateway.networking.k8s.io/v1`, Gateway API ‚â• 1.5), and the shared Gateway a ListenerSet attaches to must opt in via `spec.allowedListeners`.
- `security.libsonnet` ‚Äî the Pod Security Standards profiles (`restricted`/`baseline`/`privileged`) as composable mixins, same axis pattern as `expose`. Each profile sets **every** security knob in `config`, so the last profile composed wins and the single-knob hatches fine-tune after it. `baseline` relaxes only what `restricted` requires beyond baseline and keeps kurly's extra hardening (read-only rootfs, user namespaces); `privileged` emits no security fields at all. The ServiceAccount-token automount rule is deliberately outside every profile (it is SA hygiene, not PSS).
- `migrations.libsonnet` ‚Äî `migration(name, to, from=null, stage=null, actions=[])` builds one entry of a stageset-controller migration ladder (the serialized `[]Migration` behind `spec.migrationsSourceRef`); a ladder is a plain Jsonnet array of them. Optional fields and empty action lists are pruned. Actions are stageset-controller `Action` objects passed through **verbatim** ‚Äî kurly deliberately does not model their schema (it would drift against stageset's API).
- `main.libsonnet` ‚Äî the entry point: exposes the kinds (as functions), `expose`, `security`, `migrations`, the `list`/`listOf` helpers, and (via `+ (import './features.libsonnet')`) every feature at the top level (`kurly.store`, ‚Ä¶). `list(app)` wraps an app's manifests (including its hidden owned manifests) in a `kind: List` ‚Äî the shape a consumer renders a composed workload to; `listOf(manifests)` wraps an explicit, null-dropping set.

**The `+`-feature pattern:** a kind `function(name, image)` returns an object whose hidden `config::` holds all knobs and whose *visible* fields are the manifests, computed from `self.config`. Features are standalone `{ config+:: ‚Ä¶ }` mixins added with `+`; because manifests derive from the merged config, features late-bind regardless of order, and `std.objectValues(app)` is exactly the manifest set (hidden `ownedManifests::` rides along via `list()`). A workload is authored as `function(params) ‚Ä¶` so JaaS can feed parameters as TLAs; `jsonnet` renders the defaults for the artifact pipeline. Two traps:

- Inside a field whose value is a **nested object literal**, `self` binds to that literal, not the app ‚Äî `base.core` captures `local this = self` for those cases (e.g. `selectorLabels`), and the expose recipes capture `local app = self` for the same reason.
- A feature must contribute **only** to `config`, never patch a manifest field directly ‚Äî otherwise a later feature that recomputes that manifest from config would clobber it.

**Render-time feature exclusion:** a feature that must not co-occur with another adds its name under a shared group in `config.exclusive` (e.g. every `expose.*` recipe does `config+:: { exclusive+: { exposure+: [name] } }`); a `base.core` `assert` fails the render, naming the conflict, if any group has more than one distinct member. This is how "one exposure per workload" is enforced ‚Äî the constraint travels with the features, not a hand-written value-flag guard.

**Selector stability:** `selectorLabels` (plus the `name` label k8s-libsonnet's constructors force) feed immutable `matchLabels`; user labels from `withLabels` go to metadata + pod template only. Do not let user labels reach a selector.

**Security defaults:** every kind ships the Pod Security Standards `restricted` profile ‚Äî every setting is a `config::` knob (runAsNonRoot, seccompProfile, allowPrivilegeEscalation, dropAllCapabilities, readOnlyRootFilesystem, hostUsers) rendered pod-level via `base.core`'s `podSecurity::` fragment and container-level in `container::`. A relaxed knob **omits** its field from the manifest rather than writing the Kubernetes default explicitly ‚Äî so `security.privileged` output carries no security stanzas at all. Knobs relax two ways: the single-knob hatch features (`kurly.rootUser`, `kurly.writableRootFilesystem`, `kurly.hostUsers`) or the `kurly.security.*` profile mixins (which set all knobs ‚Äî hatches must compose *after* a profile). `automountServiceAccountToken` (only with an explicit ServiceAccount) is outside both. New workload kinds must merge `podSecurity + podVolumes` into their pod template spec and inherit `container::`; keep the default posture fully `restricted`.

## Dependency policy

`jsonnetfile.json` pins k8s-libsonnet to `main`, and `vendor/` + the lock file are **gitignored**: the JOI k8s-libsonnet image tracks upstream HEAD daily, so CI vendoring HEAD tests exactly what clusters run ‚Äî an upstream break in CI is a wanted signal. Do not commit a lock file or pin a SHA. k8s-libsonnet is **not bundled** into the OCI image; jaas supplies it at render time as its own JsonnetLibrary.

## Tests

`tests/kurly_test.jsonnet` is an assertion suite: every field is a `std.assertEqual` (raises on mismatch), and CI additionally checks all values are `true` via jq. Two negative checks live in the CI test job (`scripts/check-tests.sh`) because jsonnet has no try/catch, so error paths cannot live in the assertion suite: a worker + exposure composition must fail (`requiresService`), and composing two exposures must fail (the `exposure` exclusion group). The examples double as end-to-end fixtures: `hack/validate-examples.sh` renders each one, splits the `List` items, and validates every manifest with kubeconform `-strict` ‚Äî core kinds against the upstream Kubernetes schemas, Gateway API kinds against the community CRD schema catalog (`-ignore-missing-schemas` covers kinds absent from both, e.g. XListenerSet; watch the summary's skipped count).

## Workload artifact contract

A workload publishes its jsonnet **SOURCE**, not pre-rendered manifests ‚Äî as a **single-layer** `FROM scratch` vendor-tree image (`/github.com/metio/kurly/workloads/<name>/‚Ä¶`), the exact same build and shape as the kurly library and JOI images. JaaS pulls it (a selector-less Flux `OCIRepository`, since it is single-layer), imports a stage's composable app by its canonical path, composes the consumer's `+` features, and renders with `kurly.list`; stageset deploys the result.

**Why source, not pre-rendered:** a stage is a `function(params)` composable app, so rendering it at release time would bake placeholder defaults (`tik.example.com`, `1Gi`) that no real deployment wants ‚Äî the consumer's parameters are `+`-composed at *their* render, in JaaS. **Why single-layer, built as an image (not an `oras`-pushed tarball):** a raw `source.tar.gz` layer carries an `image.title` annotation and is consumed as a *file*, not extracted to the vendor tree ‚Äî JaaS would find no source to import; and a multi-layer artifact needs a `layerSelector`, which a selector-less `OCIRepository` and JaaS's vendor-tree search do not use.

**Maturity and the README deploy section are generated, not hand-written.** Each workload in `catalog.json` carries a `maturity` ‚Äî a derived `tier` (`rendered` ‚Üí `tested` ‚Üí `e2e`) plus an optional operator-attested `production` record. `gen-maturity` derives the tier from the repository's own signals (it greps `hack/smoke/scenario-*.sh` for `e2e` and `tests/*.jsonnet` for `tested`) into `catalog/maturity.gen.libsonnet`, and `check-catalog` regenerates + drift-checks it, so a tier can never claim more than the repo proves; production dates live in the hand-edited `catalog/production.libsonnet`. Every workload README ends with a generated, marker-delimited (`<!-- BEGIN generated: jaas-deploy -->`) section carrying the maturity badge and the full OCIRepository ‚Üí JsonnetLibrary ‚Üí JsonnetSnippet ‚Üí StageSet walkthrough; `gen-readme` splices it in from `catalog/readme.jsonnet` and `check-readme` fails if it drifts. **When adding or changing a workload, write only the README prose above that marker, then run `gen-readme`** ‚Äî never hand-edit the generated section. `tik`'s README is the hand-authored exemplar the generator is modelled on and is left untouched.

A workload carries its version as data: each stage does `local version = std.rstripChars(importstr './version.txt', '\n')`, and the dir holds a `version.txt` committed as `dev` (a data file, licensed via `REUSE.toml`, kept outside the `*.libsonnet` stage globs so no gate treats it as a stage). The release job overwrites that one file with the calver (a whole-file `printf`, no in-place edit) so rendered objects carry `app.kubernetes.io/version`, then builds the shared root **`workload.Containerfile`** ‚Äî `FROM scratch`, one `ARG WORKLOAD`, one `COPY workloads/${WORKLOAD} ‚Ä¶` (one directory ‚Üí one layer, no staging stage needed) ‚Äî via `metio/ci/container-release`. A local single-arch build first **asserts exactly one layer** and that the shipped `version.txt` carries this run's version. Editing `workload.Containerfile` re-releases every workload (the discover job treats it as a shared change), since it packages all of them.

## CI

`verify.yml` is the PR gate (fmt, test, examples, reuse, yaml, github-actions, markdown, typos, dco), ending in the single `Verify` aggregate job ‚Äî the only check to mark required in branch protection. Every tool job is checkout ‚Üí the repo-local composite `.github/actions/nix-devshell` ‚Üí `nix develop --command <gate>`: the composite installs Nix (`DeterminateSystems/nix-installer-action`) and caches the `/nix` store through the GitHub Actions cache (`nix-community/cache-nix-action`, keyed on `hashFiles(flake.nix, flake.lock)` with a prefix-fallback restore), so the devShell closure is downloaded only when the flake pin changes ‚Äî every other run restores it in seconds. The tools come from the flake's devShell, pinned by `flake.lock`, never from marketplace actions or `@latest` installs ‚Äî so CI and local shells cannot disagree on versions. The `dco` job is the one exception (pure git, no tools). `release.yml` does **multi-unit releases from one repository, modeled on metio/helm-charts**: the releasable units are the library (`main.libsonnet` + `lib/` + `Containerfile`) and every directory under `workloads/`. A `discover` job classifies this push's diff per unit (dispatch force-releases one unit or, with empty input, everything ‚Äî also the recovery path for a failed run, since change detection covers only the push's own range); all units released in one run share a single `metio/ci/calver` version, but each gets its own tag (`library-<version>`, `<workload>-<version>`), its own GitHub Release, and its own notes via `metio/ci/release-notes` scoped by **`include-paths` and `tag-pattern`** to that unit's files and tag lineage ‚Äî one unit's commits never leak into another's changelog. The workload matrix runs with `fail-fast: false` so one unit's failure never blocks another's release. The `library` job asserts **exactly one layer** and that the image **renders `kurly.http` from its own tree** (with k8s-libsonnet supplied separately, as JaaS does ‚Äî so a layer missing `lib/` fails here, not in a consumer's cluster), then publishes via `metio/ci/container-release`; each `workload` job stamps the calver into the workload's `version.txt`, asserts one layer plus the stamp on a local build, then publishes the **single-layer source image** via `metio/ci/container-release` from the shared `workload.Containerfile` (`--build-arg WORKLOAD=<name>`) ‚Äî the same single-layer discipline as the library. A workload directory must never be named `library` (tag-prefix collision; the discover job fails on it). Library changes deliberately do **not** re-release workloads ‚Äî a workload picks up library changes with its own next release or a forced dispatch, mirroring helm-charts' common-chart handling. Renovate keeps `flake.lock` fresh (the repo-local `renovate.json` enables the `nix` manager + lock-file maintenance on top of the org preset).
